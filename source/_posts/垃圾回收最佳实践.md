# Unity 垃圾回收最佳实践

## 前言

在 Unity 游戏开发中，垃圾回收（Garbage Collection，简称 GC）是影响游戏性能的关键因素之一。当 GC 触发时，游戏主线程会被暂停，导致帧率下降、卡顿等问题。特别是在移动设备上，频繁的 GC 甚至会造成明显的卡顿感。

本文基于 Unity 官方文档，详细讲解 GC 产生的原因以及优化技巧，帮助你写出高性能的游戏代码。

---

## 1. 理解 Unity 的 GC 机制

### 1.1 什么是托管堆（Managed Heap）

在 C# 中，内存分为两种：
- **栈（Stack）**：存储值类型（int、float、struct 等），自动管理
- **托管堆（Managed Heap）**：存储引用类型（class、string、array 等），由 GC 管理

当你创建一个引用类型对象时，Unity 会在托管堆上分配内存。当这个对象不再被引用时，GC 会回收这块内存。

### 1.2 Unity 的 GC 特点

**重要：Unity 的 GC 不是分代式的（Non-generational）**

.NET 的 GC 是分代式的，可以高效处理大量小对象。但 Unity 使用的是 Boehm GC，它会扫描整个托管堆，这意味着：
- GC 时间与托管堆大小成正比
- 频繁创建小对象会导致频繁的 GC
- GC 暂停时间可能很长

### 1.3 什么时候会触发 GC

- 托管堆空间不足时
- 手动调用 `System.GC.Collect()`（不推荐）
- 场景加载时

## 2. 引用类型管理

### 2.1 避免重复字符串拼接

字符串在 C# 中是**不可变的引用类型**。每次拼接字符串，都会创建一个新的字符串对象，旧的字符串对象变成垃圾。

**❌ 错误示例：在 Update 中拼接字符串**

```csharp
public class ExampleScript : MonoBehaviour
{
    public Text scoreBoard;
    public int score;

    void Update()
    {
        // 每帧都创建新字符串，产生大量 GC
        string scoreText = "Score: " + score.ToString();
        scoreBoard.text = scoreText;
    }
}
```

**✅ 正确示例：只在分数变化时更新**

```csharp
public class ExampleScript : MonoBehaviour
{
    public Text scoreBoard;
    public int score;
    private int oldScore;

    void Update()
    {
        if (score != oldScore)
        {
            string scoreText = "Score: " + score.ToString();
            scoreBoard.text = scoreText;
            oldScore = score;
        }
    }
}
```

**✅ 更优方案：完全避免字符串拼接**

```csharp
public class ExampleScript : MonoBehaviour
{
    public Text scoreBoardTitle;
    public Text scoreBoardDisplay;
    public int score;
    private int oldScore;

    void Start()
    {
        scoreBoardTitle.text = "Score: ";
    }

    void Update()
    {
        if (score != oldScore)
        {
            scoreBoardDisplay.text = score.ToString();
            oldScore = score;
        }
    }
}
```

**✅ 使用 StringBuilder 处理大量字符串拼接**

```csharp
using System.Text;

public class ExampleScript : MonoBehaviour
{
    private StringBuilder _sb = new StringBuilder(16);

    string ConcatExample(string[] stringArray)
    {
        _sb.Clear();
        for (int i = 0; i < stringArray.Length; i++)
        {
            _sb.Append(stringArray[i]);
        }
        return _sb.ToString();
    }
}
```

### 2.2 避免闭包和匿名方法

方法引用在 C# 中是引用类型，会在托管堆上分配内存。特别是**闭包**，会创建额外的匿名类实例。

**❌ 错误示例：使用闭包**

```csharp
void Update()
{
    int desiredDivisor = GetDesiredDivisor();

    // 匿名方法引用了外部变量 desiredDivisor，形成闭包
    // 每次调用都会分配内存
    listOfNumbers.Sort((x, y) =>
        (int)x.CompareTo((int)(y / desiredDivisor))
    );
}
```

**✅ 正确示例：避免使用外部变量**

```csharp
void Update()
{
    // 使用常量或成员变量，避免闭包
    listOfNumbers.Sort((x, y) =>
        (int)x.CompareTo((int)(y / 2))
    );
}
```

**✅ 更优方案：使用预定义的比较器**

```csharp
private class MyComparer : IComparer<float>
{
    private int _divisor;

    public MyComparer(int divisor)
    {
        _divisor = divisor;
    }

    public int Compare(float x, float y)
    {
        return x.CompareTo(y / _divisor);
    }
}

private MyComparer _comparer = new MyComparer(2);

void Update()
{
    listOfNumbers.Sort(_comparer);
}
```

### 2.3 避免装箱（Boxing）

**装箱**是指将值类型自动转换为引用类型。这是 Unity 项目中最常见的意外内存分配来源。

**❌ 错误示例：装箱**

```csharp
int x = 1;
object y = new object();
y.Equals(x);  // x 被装箱为 object，产生 GC
```

**❌ 常见的装箱场景：使用 ArrayList**

```csharp
// ArrayList 只接受 object，值类型会被装箱
ArrayList list = new ArrayList();
list.Add(1);      // int 被装箱
list.Add(2.5f);   // float 被装箱
```

**✅ 正确示例：使用泛型集合**

```csharp
// List<T> 是强类型，不会装箱
List<int> list = new List<int>();
list.Add(1);      // 不会产生 GC
list.Add(2);
```

**如何识别装箱：**

在 CPU Profiler 中，装箱会显示为以下形式：
- `System.Int32::Box(int32)`
- `System.Single::Box(float)`

### 2.4 避免 params 修饰符

使用 `params` 修饰符的方法会自动创建一个数组来存储参数。

**❌ 错误示例**

```csharp
// 这个方法每次调用都会创建数组
void LogMessage(params object[] args)
{
    // ...
}

void Update()
{
    LogMessage("Health:", 100);  // 每帧产生 GC
}
```

**✅ 正确示例：使用方法重载**

```csharp
// 提供不使用 params 的重载版本
void LogMessage(string msg)
{
    // ...
}

void LogMessage(string msg, int value)
{
    // ...
}
```

## 3. 数组优化

### 3.1 将数组作为参数传递

**❌ 错误示例：每次都创建新数组**

```csharp
float[] RandomList(int numElements)
{
    var result = new float[numElements];  // 每次调用都分配内存
    for (int i = 0; i < numElements; i++)
    {
        result[i] = Random.value;
    }
    return result;
}
```

**✅ 正确示例：复用数组**

```csharp
private float[] _cachedArray = new float[100];

void RandomList(float[] arrayToFill)
{
    for (int i = 0; i < arrayToFill.Length; i++)
    {
        arrayToFill[i] = Random.value;
    }
}

void Update()
{
    RandomList(_cachedArray);  // 复用数组，无 GC
}
```

### 3.2 避免重复访问返回数组的 Unity API

Unity 中很多属性（如 `mesh.vertices`）每次访问都会创建一个新的数组副本。

**❌ 错误示例：在循环中重复访问**

```csharp
void Update()
{
    // 每次访问 mesh.vertices 都会创建新数组
    for (int i = 0; i < mesh.vertices.Length; i++)
    {
        float x = mesh.vertices[i].x;  // 又创建一个新数组
        float y = mesh.vertices[i].y;  // 又创建一个新数组
        float z = mesh.vertices[i].z;  // 又创建一个新数组
        DoSomething(x, y, z);
    }
}
```

**✅ 正确示例：缓存数组**

```csharp
void Update()
{
    var vertices = mesh.vertices;  // 只创建一次数组
    for (int i = 0; i < vertices.Length; i++)
    {
        float x = vertices[i].x;
        float y = vertices[i].y;
        float z = vertices[i].z;
        DoSomething(x, y, z);
    }
}
```

**✅ 更优方案：使用 List 缓存**

```csharp
private List<Vector3> _vertices = new List<Vector3>();

void Update()
{
    mesh.GetVertices(_vertices);  // 填充现有 List，无额外分配
    for (int i = 0; i < _vertices.Count; i++)
    {
        DoSomething(_vertices[i].x, _vertices[i].y, _vertices[i].z);
    }
}
```

**常见的分配内存的 API 及其替代方案：**

| 产生 GC 的 API | 非分配替代方案 |
|----------------|----------------|
| `Physics.RaycastAll()` | `Physics.RaycastNonAlloc()` |
| `Animator.parameters` | `Animator.parameterCount` + `Animator.GetParameter()` |
| `Renderer.sharedMaterials` | `Renderer.GetSharedMaterials()` |
| `Input.touches` | `Input.touchCount` + `Input.GetTouch()` |

### 3.3 大型数组使用 NativeArray

当数组元素超过 1 万个时，GC 扫描会变慢。考虑使用 `Unity.Collections.NativeArray`。

**✅ 使用 NativeArray**

```csharp
using Unity.Collections;

private NativeArray<float> _largeArray;

void Start()
{
    _largeArray = new NativeArray<float>(10000, Allocator.Persistent);
}

void OnDestroy()
{
    if (_largeArray.IsCreated)
        _largeArray.Dispose();
}
```

## 4. 对象池

对象池通过**复用对象**来减少频繁的创建与销毁，从而降低 GC 压力和性能开销。

需要注意：

> 对象池减少的是“分配频率”，而不是完全消除 GC。

------

### 4.1 使用 Unity 内置 ObjectPool

Unity 2021+ 提供了 `UnityEngine.Pool.ObjectPool<T>`。

```c#
using UnityEngine;
using UnityEngine.Pool;

public class BulletManager : MonoBehaviour
{
    [SerializeField] private Bullet bulletPrefab;
    private ObjectPool<Bullet> _bulletPool;

    void Awake()
    {
        _bulletPool = new ObjectPool<Bullet>(
            createFunc: () => Instantiate(bulletPrefab),

            actionOnGet: (bullet) =>
            {
                bullet.gameObject.SetActive(true);
            },

            actionOnRelease: (bullet) =>
            {
                bullet.gameObject.SetActive(false);
            },

            actionOnDestroy: (bullet) =>
            {
                Destroy(bullet.gameObject);
            },

#if UNITY_EDITOR
            collectionCheck: true,
#else
            collectionCheck: false,
#endif

            defaultCapacity: 20,
            maxSize: 100
        );
    }

    public Bullet GetBullet()
    {
        return _bulletPool.Get();
    }

    public void ReturnBullet(Bullet bullet)
    {
        _bulletPool.Release(bullet);
    }
}
```

------

### 4.2 关键注意事项

- `Instantiate` 仍然会产生分配，仅在池扩容时发生
- `maxSize` 设置过小会导致频繁创建和销毁
- `Release` 只是回收对象，不会触发销毁
- `actionOnDestroy` 仅在对象被真正销毁时调用

------

### 4.3 集合类型对象池

对于临时集合（如 `List`、`Dictionary`），可以使用 `CollectionPool`。

```c#
using UnityEngine.Pool;

void Update()
{
    var list = ListPool<int>.Get();

    try
    {
        list.Add(1);
        list.Add(2);
    }
    finally
    {
        ListPool<int>.Release(list);
    }
}
```

------

### 4.4 使用注意事项

- 不要在 Release 后继续持有引用
- 不要跨帧保存池中的集合
- `CollectionPool` 内部会自动调用 `Clear()`

------

### 4.5 总结

- 对象池适用于高频创建/销毁对象（如子弹、特效）
- 能显著降低 GC 压力和 CPU 开销
- 需要合理设置容量并规范使用，否则效果有限

## 5. foreach 与 GC

### 5.1 foreach 是否会产生 GC

`foreach` 本身并不会必然产生 GC，是否产生 GC 取决于：

- 枚举器是否是值类型（struct）
- 是否被转换为接口类型（导致装箱）
- 是否使用了编译器生成的引用类型（如 yield）

------

####  会产生 GC 的情况

##### 1）遍历接口类型（常见误区）

```c#
IEnumerable<int> list = new List<int> { 1, 2, 3 };

foreach (var item in list)  // 会产生 GC
{
}
```

原因：

- `List<T>.Enumerator` 是 struct
- 但通过 `IEnumerable<T>` 调用时会被装箱
- 装箱发生在托管堆上 → 产生 GC

------

##### 2）使用 yield 返回的集合

```c#
IEnumerable<int> GetData()
{
    yield return 1;
    yield return 2;
}

foreach (var item in GetData())  // 会产生 GC
{
}
```

原因：

- 编译器生成状态机类（引用类型）
- 在堆上分配

------

##### 3）非泛型集合

```c#
ArrayList list = new ArrayList();

foreach (var item in list)  // 会产生 GC
{
}
```

原因：

- 使用的是引用类型枚举器
- 可能涉及装箱

------

#### 不产生 GC 的情况

##### 1）直接遍历 List<T>

```c#
List<int> list = new List<int> { 1, 2, 3 };

foreach (var item in list)  // 不产生 GC
{
}
```

原因：

- 使用 `List<T>.Enumerator`（struct）
- 在栈上分配，无堆分配

------

##### 2）遍历 Dictionary<K, V>

```c#
Dictionary<int, string> dict = new Dictionary<int, string>();

foreach (var kvp in dict)  // 不产生 GC
{
}
```

------

##### 3）遍历数组

```c#
int[] array = { 1, 2, 3 };

foreach (var item in array)  // 不产生 GC
{
}
```

原因：

- 编译器通常会优化为 for 循环

------

### 5.2 常见误解

很多资料会说：

> “foreach 遍历 List 会产生 GC”

这是不准确的。真正的问题通常是下面这种写法：

```c#
IReadOnlyList<int> list = new List<int>();

foreach (var item in list)  // 可能产生 GC
{
}
```

问题不在 `foreach`，而在于：

- 使用了接口类型
- 导致 struct 枚举器被装箱

------

### 5.3 for 与 foreach 对比

| 遍历方式 | List<T> | Dictionary | 数组  | IEnumerable |
| -------- | ------- | ---------- | ----- | ----------- |
| foreach  | 无 GC   | 无 GC      | 无 GC | 有 GC       |
| for      | 无 GC   | 不支持     | 无 GC | 不支持      |

------

### 5.4 是否应该用 for 替代 foreach

不需要一刀切。

建议：

- 在性能敏感代码（如 Update、AI、频繁调用逻辑）中：

  ```c#
  for (int i = 0; i < list.Count; i++)
  {
      var item = list[i];
  }
  ```

- 在普通逻辑中：

  ```c#
  foreach (var item in list)
  {
  }
  ```

------

### 5.5 关于 Span<T>

```c#
Span<int> span = array;
```

特点：

- 不分配堆内存
- 适用于数组等连续内存

但在 Unity 中：

- 使用场景有限
- 更多用于底层优化，而不是日常遍历替代

------

### 5.6 总结

- `foreach` 本身不会产生 GC
- 关键在于是否发生装箱或使用引用类型枚举器
- 避免在高频代码中使用接口类型进行遍历
- 在性能关键路径中优先使用 `for` 循环以保证稳定性

## 6. 其他最佳实践

### 6.1 缓存组件引用

**❌ 错误示例：每次都获取组件**

```csharp
void Update()
{
    // 每帧都调用 GetComponent，产生 GC
    var renderer = GetComponent<Renderer>();
    renderer.material.color = Color.red;
}
```

**✅ 正确示例：缓存组件**

```csharp
private Renderer _cachedRenderer;

void Awake()
{
    _cachedRenderer = GetComponent<Renderer>();
}

void Update()
{
    _cachedRenderer.material.color = Color.red;  // 无 GC
}
```

### 6.2 使用 CompareTag

**❌ 错误示例：使用字符串比较**

```csharp
void OnTriggerEnter(Collider other)
{
    if (other.tag == "Enemy")  // 产生 GC
    {
        // ...
    }
}
```

**✅ 正确示例：使用 CompareTag**

```csharp
void OnTriggerEnter(Collider other)
{
    if (other.CompareTag("Enemy"))  // 无 GC
    {
        // ...
    }
}
```

### 6.3 避免在 Update 中分配内存

**❌ 错误示例：每帧创建 List**

```csharp
void Update()
{
    List<float> nearestNeighbors = new List<float>();  // 每帧分配
    FindDistances(nearestNeighbors);
    nearestNeighbors.Sort();
}
```

**✅ 正确示例：复用 List**

```csharp
private List<float> _nearestNeighbors = new List<float>();

void Update()
{
    _nearestNeighbors.Clear();  // 清空但保留内存
    FindDistances(_nearestNeighbors);
    _nearestNeighbors.Sort();
}
```

### 6.4 协程的 GC 问题

在 Unity 中，协程（Coroutine）常用于处理延时、分帧逻辑等。但在性能敏感场景中，需要注意：

> 协程通常会产生 GC 分配。

------

#### 6.4.1 协程产生 GC 的根本原因

##### 1）IEnumerator 会被编译为状态机类

示例代码：

```c#
IEnumerator DoSomething()
{
    yield return null;
}
```

编译后会被转换为类似如下结构（简化）：

```c#
class DoSomethingStateMachine : IEnumerator
{
    int state;
    object current;

    public bool MoveNext()
    {
        // 状态机逻辑
    }
}
```

关键点：

- 这是一个 **class（引用类型）**
- 每次调用协程方法，都会创建一个新的状态机对象

因此：

```c#
StartCoroutine(DoSomething());
```

实际会发生：

- 创建一个状态机实例（堆分配）
- → 产生 GC

------

##### 2）yield return 可能产生额外分配

例如：

```c#
yield return new WaitForSeconds(1f);
```

每次执行都会：

- 创建一个新的 `WaitForSeconds` 对象
- 分配在托管堆上

------

#### 6.4.2 常见 GC 来源总结

| 来源                              | 是否产生 GC | 说明             |
| --------------------------------- | ----------- | ---------------- |
| 协程方法（IEnumerator）           | 是          | 状态机是引用类型 |
| `yield return new WaitForSeconds` | 是          | 每次 new         |
| `yield return null`               | 否          | Unity 内部复用   |
| 高频调用 `StartCoroutine`         | 是          | 持续创建状态机   |

------

#### 6.4.3 典型问题场景

##### ❌ 在高频逻辑中反复启动协程

```c#
void Update()
{
    StartCoroutine(DoSomething()); // 每帧调用
}
```

问题：

- 每帧都会创建状态机对象
- 如果包含 `WaitForSeconds`，还会额外分配
- → 持续 GC → 可能引发卡顿

------

#### 6.4.4 优化建议

##### ✅ 1）缓存 WaitForSeconds

```c#
private static readonly WaitForSeconds wait = new WaitForSeconds(1f);

IEnumerator DoSomething()
{
    yield return wait;
}
```

避免重复创建对象。

------

##### ✅ 2）避免高频启动协程

使用状态控制：

```c#
bool isRunning = false;

void Update()
{
    if (!isRunning)
    {
        StartCoroutine(DoSomething());
        isRunning = true;
    }
}
```

------

##### ✅ 3）简单逻辑用 Update 替代

```c#
float timer;

void Update()
{
    timer += Time.deltaTime;
    if (timer >= 1f)
    {
        timer = 0;
        // 执行逻辑
    }
}
```

特点：

- 无堆分配
- 更可控

------

##### ✅ 4）使用 UniTask（可选）

```c#
async UniTaskVoid DoSomethingAsync()
{
    await UniTask.Delay(1000);
}
```

说明：

- 基于 struct 的异步实现
- 大多数情况下避免堆分配
- 适合替代部分协程逻辑

> 注意：并非绝对零 GC，但显著减少分配

------

#### 6.4.5 总结

- 协程的 GC 主要来自：
  - `IEnumerator` 状态机对象（堆分配）
  - `yield return` 中创建的对象
- 协程本身不是问题，问题在于**使用方式**
- 在高频逻辑中，应避免频繁创建协程

------

#### 一句话总结

> 协程产生 GC 的本质原因，是 C# 会将 `IEnumerator` 编译为堆分配的状态机对象，同时 `yield return` 中的对象也可能带来额外分配。

## 7. 性能分析工具

### 7.1 Unity Profiler

使用 Unity Profiler 分析 GC：

1. 打开 **Window > Analysis > Profiler**
2. 选择 **Memory** 模块
3. 查看 **GC Alloc** 列，找出分配内存的代码

### 7.2 查找装箱

在 Profiler 的 **CPU** 模块中，搜索以下方法名：
- `System.Int32::Box`
- `System.Single::Box`
- `System.Object::Box`

### 7.3 使用 IL 查看器

使用 Rider、ReSharper 或 dotPeek 查看 IL 代码，搜索 `box` 指令来定位装箱。

---

## 8. 总结

### 优化优先级

1. **避免在 Update 中分配内存**（最高优先级）
2. **使用对象池**复用 GameObject 和集合
3. **避免字符串拼接**，特别是每帧的拼接
4. **使用 for 循环**替代 foreach 遍历 List 和 Dictionary
5. **缓存组件引用**和 API 调用结果
6. **使用 CompareTag**替代字符串比较
7. **避免装箱**，使用泛型集合

### 核心原则

> **在性能关键的代码中（Update、FixedUpdate、LateUpdate），尽量避免任何内存分配。**

记住：GC 优化的目标不是完全消除 GC，而是减少 GC 的频率和影响。在合适的时机进行优化，避免过早优化。
