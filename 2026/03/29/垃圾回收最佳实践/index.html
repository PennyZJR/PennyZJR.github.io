Unity 垃圾回收最佳实践

前言

在 Unity 游戏开发中，垃圾回收（Garbage Collection，简称 GC）是影响游戏性能的关键因素之一。当 GC 触发时，游戏主线程会被暂停，导致帧率下降、卡顿等问题。特别是在移动设备上，频繁的 GC 甚至会造成明显的卡顿感。

本文基于 Unity 官方文档，详细讲解 GC 产生的原因以及优化技巧，帮助你写出高性能的游戏代码。

1. 理解 Unity 的 GC 机制

1.1 什么是托管堆（Managed Heap）

在 C# 中，内存分为两种：

  • 栈（Stack）：存储值类型（int、float、struct 等），自动管理
  • 托管堆（Managed Heap）：存储引用类型（class、string、array 等），由 GC 管理

当你创建一个引用类型对象时，Unity 会在托管堆上分配内存。当这个对象不再被引用时，GC 会回收这块内存。

1.2 Unity 的 GC 特点

重要：Unity 的 GC 不是分代式的（Non-generational）

.NET 的 GC 是分代式的，可以高效处理大量小对象。但 Unity 使用的是 Boehm GC，它会扫描整个托管堆，这意味着：

  • GC 时间与托管堆大小成正比
  • 频繁创建小对象会导致频繁的 GC
  • GC 暂停时间可能很长

1.3 什么时候会触发 GC

  • 托管堆空间不足时
  • 手动调用 System.GC.Collect()（不推荐）
  • 场景加载时

4. 对象池

对象池通过复用对象来减少频繁的创建与销毁，从而降低 GC 压力和性能开销。

需要注意：

对象池减少的是“分配频率”，而不是完全消除 GC。

4.1 使用 Unity 内置 ObjectPool

Unity 2021+ 提供了 UnityEngine.Pool.ObjectPool<T>。

4.2 关键注意事项

  • Instantiate 仍然会产生分配，仅在池扩容时发生
  • maxSize 设置过小会导致频繁创建和销毁
  • Release 只是回收对象，不会触发销毁
  • actionOnDestroy 仅在对象被真正销毁时调用

4.3 集合类型对象池

对于临时集合（如 List、Dictionary），可以使用 CollectionPool。

4.4 使用注意事项

  • 不要在 Release 后继续持有引用
  • 不要跨帧保存池中的集合
  • CollectionPool 内部会自动调用 Clear()

4.5 总结

  • 对象池适用于高频创建/销毁对象（如子弹、特效）
  • 能显著降低 GC 压力和 CPU 开销
  • 需要合理设置容量并规范使用，否则效果有限

5. foreach 与 GC

5.1 foreach 是否会产生 GC

foreach 本身并不会必然产生 GC，是否产生 GC 取决于：

  • 枚举器是否是值类型（struct）
  • 是否被转换为接口类型（导致装箱）
  • 是否使用了编译器生成的引用类型（如 yield）

会产生 GC 的情况

1）遍历接口类型（常见误区）

原因：

  • List<T>.Enumerator 是 struct
  • 但通过 IEnumerable<T> 调用时会被装箱
  • 装箱发生在托管堆上 → 产生 GC
2）使用 yield 返回的集合

原因：

  • 编译器生成状态机类（引用类型）
  • 在堆上分配
3）非泛型集合

原因：

  • 使用的是引用类型枚举器
  • 可能涉及装箱

不产生 GC 的情况

1）直接遍历 List

原因：

  • 使用 List<T>.Enumerator（struct）
  • 在栈上分配，无堆分配
2）遍历 Dictionary<K, V>
3）遍历数组

原因：

  • 编译器通常会优化为 for 循环

很多资料会说：

“foreach 遍历 List 会产生 GC”

特点：

  • 不分配堆内存

5.6 总结

  • foreach 本身不会产生 GC
  • 关键在于是否发生装箱或使用引用类型枚举器
  • 避免在高频代码中使用接口类型进行遍历
  • 在性能关键路径中优先使用 for 循环以保证稳定性

7. 性能分析工具

7.1 Unity Profiler

使用 Unity Profiler 分析 GC：

  1. 打开 Window > Analysis > Profiler
  2. 选择 Memory 模块
  3. 查看 GC Alloc 列，找出分配内存的代码

7.2 查找装箱

在 Profiler 的 CPU 模块中，搜索以下方法名：

  • System.Int32::Box
  • System.Single::Box
  • System.Object::Box

7.3 使用 IL 查看器

使用 Rider、ReSharper 或 dotPeek 查看 IL 代码，搜索 box 指令来定位装箱。

8. 总结

优化优先级

  1. 避免在 Update 中分配内存（最高优先级）
  2. 使用对象池复用 GameObject 和集合
  3. 避免字符串拼接，特别是每帧的拼接
  4. 使用 for 循环替代 foreach 遍历 List 和 Dictionary
  5. 缓存组件引用和 API 调用结果
  6. 使用 CompareTag替代字符串比较
  7. 避免装箱，使用泛型集合

核心原则

在性能关键的代码中（Update、FixedUpdate、LateUpdate），尽量避免任何内存分配。

记住：GC 优化的目标不是完全消除 GC，而是减少 GC 的频率和影响。在合适的时机进行优化，避免过早优化。
